SET 1: BASIC UNDERSTANDING QUESTIONS
================================================================================

Q1: "What is CODEBUDDY in one sentence?"
A1: CODEBUDDY is an AI-powered multi-agent coding assistant that uses 8
    specialized AI agents to help developers write, debug, review, test,
    document, and understand code through a beautiful web interface.

Q2: "What problem does CODEBUDDY solve?"
A2: Developers spend 30-50% of their time on non-coding tasks like
    debugging, reviewing, writing tests, and documenting code. CODEBUDDY
    automates all of these with specialized agents, saving hours per week.

Q3: "Who is your target user?"
A3: Software developers, CS students, and coding bootcamp learners. Anyone
    who writes code and wants to be more productive or learn faster.

Q4: "What technology stack did you use?"
A4: Backend: Python FastAPI with LangGraph for AI orchestration, SQLAlchemy
    for database, WebSocket for real-time communication. Frontend: React 19
    with TypeScript, Monaco Editor (VS Code's editor), xterm.js for terminal,
    Zustand for state management. AI: Ollama for local LLM inference with
    OpenAI cloud fallback.

Q5: "How long did it take to build?"
A5: We built the entire application during the hackathon, leveraging our
    pre-existing knowledge of the technologies. The architecture was designed
    to be modular so each team member could work on different parts in parallel.

================================================================================
SET 2: TECHNICAL ARCHITECTURE QUESTIONS
================================================================================

Q6: "How does the multi-agent system work?"
A6: We use LangGraph's StateGraph to orchestrate 8 specialized agents. Each
    agent is a node in a directed graph. When a user makes a request, the
    orchestrator analyzes the intent and selects the appropriate agent(s).
    The agents can be chained - for example, code can go from Coder to
    Reviewer to Tester. The graph handles the routing, state passing, and
    error recovery between agents.

Q7: "Why 8 separate agents instead of one big AI?"
A7: Specialization beats generalization. A debugging-focused agent with a
    debugging-optimized system prompt produces better debugging results than
    a generic model. Each agent has its own personality, expertise, and
    approach. The Coder writes clean code, the Debugger hunts for bugs, the
    Reviewer catches security issues - they're like a real dev team.

Q8: "How do agents communicate with each other?"
A8: Through the LangGraph state graph. Each agent receives the conversation
    history and can add its response. The orchestrator manages the flow - it
    can run agents sequentially (Coder then Reviewer) or in parallel (Coder
    and Architect simultaneously). The state is passed through graph edges.

Q9: "What LLM models do you support?"
A9: Three providers: 1) Ollama (local, private, free) - supports any model
    like gemma3:1b, llama3, codellama, 2) OpenAI (cloud, powerful) - GPT-4,
    GPT-3.5, 3) Built-in mock responses for demos and offline use. The
    system auto-detects which provider is available and falls back gracefully.

Q10: "How do you handle LLM failures?"
A10: Three-tier fallback: First try the configured provider (Ollama). If that
     fails, try the alternate endpoint (chat -> generate for Ollama). If
     everything fails, use built-in mock responses. This ensures the demo
     always works, which is critical for hackathon presentations.

SET 3: FEATURE DEEP-DIVE QUESTIONS
================================================================================

Q11: "Explain the code visualization feature."
A11: Our visualization engine parses Python code using the AST (Abstract
     Syntax Tree) module, automatically detects the algorithm type (sorting,
     searching, recursion, trees, graphs, etc.), and generates a step-by-step
     execution trace. The frontend renders this as an animated visualization
     with array highlighting, variable tracking, and data structure state
     at each step. Users can see exactly how bubble sort swaps elements or
     how binary search narrows the search range.

Q12: "How does the memory system work?"
A12: We built a TF-IDF based vector store using numpy. When the AI generates
     useful context, it's saved with an embedding vector. On subsequent
     queries, the system embeds the query, computes cosine similarity against
     all stored vectors, and returns the most relevant memories. This gives
     the AI "memory" across sessions - it remembers past code, decisions,
     and user preferences.

Q13: "What is the browser terminal?"
A13: We implemented a WebSocket-based terminal using xterm.js on the frontend
     and asyncio subprocesses on the backend. Users can run commands like
     ls, python, git directly in the browser. Commands are validated against
     a whitelist and shell operators are blocked for security. Output is
     streamed back in real-time.

Q14: "How do you ensure code execution safety?"
A14: Two layers: 1) Terminal: Commands go through a whitelist check. Shell
     operators (&&, ||, ;, |) are blocked. Only safe commands like ls, python,
     git are allowed. 2) Code runner: User Python code executes with
     restricted builtins - __import__, exec, eval, open, and breakpoint are
     removed. This prevents code from accessing the filesystem or importing
     dangerous modules.

Q15: "What are the 8 agents and their specific roles?"
A15: 1) Coder - Writes clean, functional code from descriptions
     2) Debugger - Analyzes code for bugs, suggests fixes with explanations
     3) Architect - Designs system architecture, component relationships
     4) Reviewer - Code review covering security, performance, style
     5) Documenter - Generates docs, API references, README files
     6) Tester - Creates unit tests, integration tests, edge cases
     7) Refactorer - Optimizes code, reduces complexity, improves patterns
     8) Explainer - Teaches code logic, complexity analysis, concepts

SET 4: DIFFERENTIATION & COMPETITION QUESTIONS
================================================================================

Q16: "How is this different from GitHub Copilot?"
A16: Three key differences: 1) Specialized agents vs one generic model -
     Copilot is one model for everything; we have 8 specialists. 2) Visual
     code tracing - Copilot shows code completion; we show step-by-step
     algorithm execution. 3) Multi-agent chaining - our agents collaborate
     (Coder -> Reviewer -> Tester). 4) Privacy - runs locally with Ollama,
     no code leaves your machine.

Q17: "How is this different from ChatGPT/Claude?"
A17: 1) Purpose-built for code - our agents have code-specific system prompts
     and workflows. 2) Multi-agent collaboration - ChatGPT is one model; we
     chain specialists. 3) Visual tracing - no LLM shows step-by-step code
     execution. 4) Local deployment - runs on your machine with Ollama.
     5) Persistent memory - remembers context across sessions.

Q18: "What's your unfair advantage?"
A18: The visualization engine. No other tool auto-detects algorithms and
     shows step-by-step execution traces. Combined with multi-agent
     orchestration and local LLM support, CODEBUDDY is a complete
     developer productivity platform, not just another chatbot.

Q19: "What would competitors need to replicate?"
A19: They'd need to build: 1) A multi-agent orchestration system, 2) An
     AST-based code analysis and visualization engine, 3) A vector memory
     store, 4) A WebSocket streaming infrastructure, 5) Fallback-resilient
     LLM integration. Our modular architecture makes each piece replaceable,
     but building the whole stack is a significant effort.

SET 5: SCALABILITY & PRODUCTION QUESTIONS
================================================================================

Q20: "Is this production-ready?"
A20: The architecture is production-quality: async FastAPI for high concurrency,
     WebSocket streaming for real-time UX, proper error handling with fallbacks,
     clean separation of concerns. For true production: we'd add container-based
     sandboxing, JWT authentication, PostgreSQL, Redis caching, and horizontal
     scaling. The hackathon version demonstrates all the core concepts.

Q21: "How does this scale?"
A21: FastAPI handles thousands of concurrent WebSocket connections. The agent
     system is stateless between requests, so it scales horizontally. SQLite
     can be swapped to PostgreSQL for production. Ollama can be scaled with
     load balancing. The frontend is a static React build, easily served from
     a CDN.

Q22: "How do you handle concurrent users?"
A22: FastAPI's async architecture handles concurrency natively. Each WebSocket
     connection is an async coroutine. The LLM service uses httpx async
     clients with connection pooling. Database sessions are created per-request
     and auto-closed. The broadcast system sends to all connections
     concurrently using asyncio.gather.

Q23: "What about security in production?"
A23: We'd add: 1) JWT-based authentication with refresh tokens, 2) Rate
     limiting per user/IP, 3) Container sandboxing (Docker/nsjail) for code
     execution, 4) Input validation with Pydantic, 5) CORS configuration
     for allowed origins, 6) HTTPS enforcement, 7) Secret management via
     environment variables.

SET 7: CHALLENGE & EDGE CASE QUESTIONS
================================================================================

Q28: "What if the LLM gives wrong code?"
A28: That's why we have the Reviewer and Tester agents. The Reviewer catches
     potential issues, and the Tester validates the code works. The user can
     also see the visualization trace to understand what the code actually
     does. We're building a checks-and-balances system, not blind trust.

Q29: "What about hallucinations?"
A29: Multiple safeguards: 1) The code runner actually executes Python code
     and catches errors. 2) The Tester agent creates and runs tests. 3) The
     Reviewer checks for common issues. 4) The Explainer helps users
     understand and verify the logic. 5) Visual traces show actual execution.

Q30: "What if someone tries to run malicious code?"
A30: Two defense layers: Terminal blocks dangerous commands via whitelist
     and shell operator detection. Code runner removes dangerous builtins
     (__import__, exec, eval, open). For production, we'd add Docker-based
     sandboxing with resource limits for complete isolation.

Q31: "What happens if Ollama is slow or unavailable?"
A31: The system degrades gracefully: 1) If Ollama is slow, requests timeout
     after 120 seconds. 2) If Ollama is unavailable, falls back to mock
     responses instantly. 3) The user always gets a response - never a
     blank screen or error. This is the 3-tier fallback architecture.

Q32: "Can you handle very large codebases?"
A32: The visualization engine works on individual code snippets (up to ~500
     lines). The chat system handles full conversations. For large codebases,
     the file management system lets users select specific files. We'd add
     AST-based code chunking and RAG for true large-codebase support.

SET 8: IMPLEMENTATION DETAIL QUESTIONS
================================================================================

Q33: "Walk me through the data flow."
A33: User types message -> Frontend sends via WebSocket -> FastAPI receives ->
     Orchestrator selects agent(s) -> Agent constructs prompt with system
     prompt + context + memory -> LLM generates response (streamed) ->
     Response chunks sent back via WebSocket -> Frontend renders in real-time
     -> Memory service saves useful context for future reference.

Q34: "How does the LangGraph integration work?"
A34: Each agent is a LangGraph node (async function). The StateGraph defines
     edges between nodes. We compile the graph and cache it per agent
     combination. The graph state contains the message history. When invoked,
     the graph routes through the appropriate nodes, passing state between
     them. Compiled graphs are cached for performance.

Q35: "How is the frontend structured?"
A35: React 19 with TypeScript, Vite for build. Pages: Welcome (video intro),
     Dashboard (agent overview), Chat (conversations), Projects, Code
     Visualization, Terminal. Components: Monaco Editor, Markdown Renderer,
     Memory Panel. State: Zustand store for global state. Routing: React
     Router v7 with nested routes.

Q36: "Explain the streaming implementation."
A36: Backend: FastAPI StreamingResponse or WebSocket. When LLM generates
     tokens, they're yielded as chunks through an async generator. Frontend:
     WebSocket connection receives JSON chunks with type "chunk" and content.
     The chat component appends each chunk to the displayed message, giving
     a typing effect. The "done" message signals completion.

Q37: "How does the algorithm detection work?"
A37: The trace_generator.py analyzes code using pattern matching on the AST
     and string content. It looks for keywords: "bubble"/"swap" for bubble
     sort, "binary"/"search" for binary search, "factorial"/"fibonacci" for
     recursion, "node"/"next" for linked lists, "bfs"/"dfs" for graphs. Once
     detected, it generates type-specific trace data for visualization.

SET 9: TEAM & PROCESS QUESTIONS
================================================================================

Q38: "How did you divide the work?"
A38: Modular architecture allowed parallel development: Backend team built
     the FastAPI server, LangGraph orchestration, and all 8 agents. Frontend
     team built the React UI, Monaco Editor integration, and visualizations.
     We used API contracts (OpenAPI/Swagger) to ensure frontend-backend
     compatibility.

Q39: "What was the hardest part?"
A39: The LangGraph integration - getting agents to chain properly while
     maintaining streaming was complex. Also, the code visualization engine
     required building an algorithm detection system from scratch. Making
     everything work together with proper error handling was challenging.

Q40: "What would you do differently?"
A40: 1) Start with authentication from day one. 2) Use a real embedding model
     instead of TF-IDF from the start. 3) Build Docker sandboxing earlier.
     4) Add integration tests for each agent. 5) Write more documentation
     during development, not after.
