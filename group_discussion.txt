SECTION 4: TECHNICAL DEEP DIVE (For judges who ask "how?")
================================================================================

Q: How does the multi-agent system work?
A: We use LangGraph's StateGraph. Each agent is a node in a directed graph.
   The orchestrator analyzes the user request, determines which agent(s) to
   invoke, and executes them in sequence or parallel. The graph edges define
   the flow - e.g., code goes Coder -> Reviewer -> Tester.

Q: How do you handle LLM failures?
A: Three-tier fallback: 1) Try the configured provider (Ollama/OpenAI),
   2) If that fails, try the alternate endpoint (chat -> generate for Ollama),
   3) If everything fails, use our built-in mock responses so the demo
   always works. This is critical for hackathon demos.

Q: How does code visualization work?
A: Our trace_generator.py parses Python code using the AST module, detects
   the algorithm type (sorting, searching, recursion, etc.), and generates
   step-by-step trace data. The frontend renders this with highlighting,
   variable tracking, and data structure visualization.

Q: How does the memory system work?
A: We have a TF-IDF based vector store using numpy. When code or context is
   saved, it's embedded into a vector. On search, the query is embedded and
   cosine similarity finds the most relevant memories. The vector store is
   backed by SQLite for persistence.

Q: How do you ensure security?
A: The terminal uses a whitelist of allowed commands with shlex-based parsing.
   Shell operators (&&, ||, ;, |) are blocked to prevent injection. The code
   runner sandboxes execution by removing dangerous builtins (__import__, exec,
   eval, open, breakpoint). User code runs with restricted builtins.

SECTION 6: COMMON HACKATHON JUDGE QUESTIONS & ANSWERS
================================================================================

Q: "What problem does this solve?"
A: Developers spend 30-50% of their time on tasks beyond just coding:
   debugging, reviewing, documenting, testing. CODEBUDDY automates all
   of these with specialized AI agents, saving hours per week.

Q: "How is this different from GitHub Copilot / ChatGPT?"
A: Three key differences:
   1) Specialized agents vs one generic model - each agent is optimized
      for its specific task
   2) Visual code tracing - no tool shows step-by-step algorithm execution
   3) Multi-agent chaining - agents can collaborate (Coder -> Reviewer -> Tester)
   4) Runs locally with Ollama - no data leaves your machine

Q: "What's the biggest technical challenge?"
A: Coordinating multiple AI agents through LangGraph while maintaining
   real-time streaming. Each agent has different inputs/outputs, and the
   orchestrator needs to chain them correctly while streaming results to
   the frontend without blocking.

Q: "What would you do with more time?"
A: 1) Container-based sandboxing for code execution (Docker/nsjail)
   2) Real embedding model (sentence-transformers) for better memory search
   3) Authentication and multi-user support
   4) Plugin system for community-contributed agents
   5) VS Code extension integration

Q: "Is this production-ready?"
A: The architecture is production-quality (async FastAPI, WebSocket streaming,
   proper error handling, fallback chains). For true production, we'd add:
   container sandboxing, proper auth, PostgreSQL, Redis caching, and
   horizontal scaling.

Q: "How do you handle edge cases?"
A: Every LLM call has a 3-tier fallback (primary -> secondary -> mock).
   The code runner catches syntax errors before execution. The terminal
   blocks dangerous commands. The frontend has error boundaries at every
   component level. We handle timeouts, disconnections, and malformed input.

Q: "What's the scalability story?"
A: FastAPI is async-first and handles thousands of concurrent connections.
   WebSocket connections are lightweight. The SQLite database can be swapped
   to PostgreSQL. Ollama can be scaled horizontally. The agent system is
   stateless between requests.
