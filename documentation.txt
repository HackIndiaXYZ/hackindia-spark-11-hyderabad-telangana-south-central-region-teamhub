1. PROJECT OVERVIEW
================================================================================

CODEBUDDY is an AI-powered multi-agent coding assistant that uses local LLMs
(via Ollama) to provide 8 specialized coding agents. It features:

  - 8 AI agents: Coder, Debugger, Architect, Reviewer, Documenter, Tester,
    Refactorer, Explainer
  - LangGraph-powered orchestrator for multi-agent workflows
  - FastAPI backend with async SQLite database
  - React 19 + TypeScript frontend with premium dark emerald + gold theme
  - WebSocket support for real-time communication
  - Fully offline operation using local Ollama models

Tech Stack:
  - Backend: Python 3.14, FastAPI, SQLAlchemy, LangChain, LangGraph, Ollama
  - Frontend: React 19, TypeScript, Vite, Zustand, React Router v7
  - Database: SQLite (via aiosqlite + SQLAlchemy async)
  - LLM: Ollama (default model: gemma3:1b)

3. ARCHITECTURE
================================================================================

The application follows a three-tier architecture:

  [React Frontend] <--HTTP/WS--> [FastAPI Backend] <--LLM--> [Ollama]

  - Frontend (port 5173): React SPA with React Router v7 for navigation,
    Zustand for state management, Monaco editor for code editing.

  - Backend (port 8000): FastAPI serving REST endpoints and WebSocket
    connections. Uses SQLAlchemy async ORM with SQLite.

  - LLM Layer: LangChain BaseChatModel subclasses communicate with Ollama
    (default: gemma3:1b). Fallback and OpenAI providers also supported.

Flow:
  1. User sends a request via Chat or Agent page
  2. Backend API receives request
  3. Agent orchestrator (LangGraph) routes to appropriate agent(s)
  4. Agent uses BaseAgent.execute() -> llm_service.chat() -> Ollama
  5. Response streamed back to frontend

4. BACKEND COMPONENTS
================================================================================

4.1 main.py
  - Creates FastAPI app with CORS middleware
  - Registers all API routers under /api/v1/
  - Health check at /health and root at /
  - WebSocket endpoint at /ws/{client_id}

4.2 Agents (app/agents/)
  - BaseAgent: Abstract base class with execute() method
    - execute() builds messages, calls llm_service.chat(), returns
      {"content": str, "tokens_used": int}
  - 8 specialized agents inherit BaseAgent:
    - CoderAgent, DebuggerAgent, ArchitectAgent, ReviewerAgent,
      DocumenterAgent, TesterAgent, RefactorerAgent, ExplainerAgent
    Each defines: name, description, icon, color, system_prompt
  - AgentOrchestrator (orchestrator.py):
    - Uses LangGraph StateGraph for multi-agent orchestration
    - Builds dynamic graph per agent subset (cached by tuple key)
    - Nodes: intent_router, planner, context_builder, agent nodes,
      reviewer, response_formatter
  - AGENT_REGISTRY (registry.py): dict of all agents keyed by "agent_key"

4.3 API (app/api/v1/)
  - agents.py: GET /agents (list), POST /agents/{agent_key}/execute
    (single agent), POST /agents/orchestrate (multi-agent)
  - chat.py: POST /chat (send message, optionally stream)
  - auth.py: POST /auth/register, /auth/login, /auth/me
  - projects.py: CRUD for projects
  - files.py: CRUD for project files
  - terminal.py: POST /terminal/execute
  - memory.py: GET/POST/DELETE memory entries
  - visualization.py: POST /visualization/generate

4.4 Services (app/services/)
  - llm_service.py: LLM integration with LangChain
  - agent_orchestrator.py: Wraps LangGraph orchestrator for API use
  - memory_service.py: Memory CRUD operations
  - websocket_manager.py: WebSocket connection management

4.5 Database (app/core/)
  - database.py: Async SQLAlchemy engine with SQLite
  - Models: User, Project, File, Chat, Memory

  -----------------------------------------------

6.2 backend/app/services/llm_service.py  <-- PRIMARY LANGCHAIN USAGE
  -----------------------------------------------
  Imports:
    from langchain_core.language_models.chat_models import BaseChatModel
    from langchain_core.messages import HumanMessage, SystemMessage
    from langchain_core.outputs import ChatGeneration, ChatResult

  Three custom chat models extending BaseChatModel:
    a) FallbackChatModel  - Returns mock responses for testing
    b) OllamaChatModel    - Calls Ollama API (default provider)
    c) OpenAIChatModel    - Calls OpenAI-compatible API

  LLMService class methods:
    - get_chat_model()       -> Returns appropriate BaseChatModel instance
    - _check_ollama()        -> Pings Ollama to verify availability
    - _call_ollama()         -> Sends request to Ollama /api/chat endpoint
    - _call_openai()         -> Sends request to OpenAI API
    - _fallback()            -> Returns mock LLM response
    - chat()                 -> Main chat method, routes to selected provider
    - chat_with_agents()     -> Multi-turn chat with agent context
  -----------------------------------------------

7. LANGGRAPH USAGE (DIRECTORIES & FILES)
================================================================================

LangGraph is used for multi-agent orchestration in the Backend layer.
Here are the exact directories and files:

7.1 backend/requirements.txt (line 19)
  -----------------------------------------------
    langgraph>=0.2.0
  -----------------------------------------------

7.2 backend/app/agents/orchestrator.py  <-- PRIMARY LANGGRAPH USAGE
  -----------------------------------------------
  Import:
    from langgraph.graph import StateGraph, START, END

  Key components:
    a) AgentState (TypedDict):
       - request: str
       - context: dict
       - agents_to_run: list[str]
       - results: dict
       - errors: list
       - execution_plan: str
       - memory_context: str
       - reviewed: bool
       - formatted_output: str

    b) _build_graph_for_agents(agent_keys: tuple) -> CompiledStateGraph:
       - Builds a StateGraph dynamically for the given agent subset
       - Caches graphs by (agent_key_tuple) for reuse
       - Standard nodes: intent_router, planner, context_builder,
         reviewer, response_formatter
       - Dynamic agent nodes: prefixed with "agent_" to avoid name
         collisions (e.g., avoid "reviewer" clash with agent name)
       - Uses closures (make_node) so each agent node runs its own
         captured agent instance at graph build time

    c) AgentOrchestrator class:
       - execute_single(request, context, agent_key) -> dict
         Routes to single agent directly (bypasses graph)
       - orchestrate(request, context, agent_keys) -> dict
         Builds/retrieves cached graph, runs with agent subset

    d) Singleton instance:
       agent_orchestrator = AgentOrchestrator()
  -----------------------------------------------

7.3 backend/app/services/agent_orchestrator.py
  -----------------------------------------------
  Import:
    from app.agents.orchestrator import agent_orchestrator as langgraph_orchestrator

  Usage:
    - execute_single() -> calls langgraph_orchestrator.execute_single()
    - orchestrate()    -> calls langgraph_orchestrator.orchestrate()
  -----------------------------------------------

7.4 Summary of LangGraph file locations:

  | Directory                        | File                   | Usage                    |
  |----------------------------------|------------------------|--------------------------|
  | backend/                         | requirements.txt       | Package dependency       |
  | backend/app/agents/              | orchestrator.py        | Graph definition + exec |
  | backend/app/services/            | agent_orchestrator.py  | Service wrapper          |

  No other files in the project import from langgraph.

8.2 Available Ollama Models
  - qwen3:8b         (5.2 GB)
  - gemma3           (3.3 GB)
  - gemma3:1b        (815 MB)   <-- default
  - nomic-embed-text (274 MB)

8.3 Provider Switching
  Set MODEL_PROVIDER in .env to switch:
  - "ollama"  -> Uses Ollama (local, default gemma3:1b)
  - "openai"  -> Uses OpenAI-compatible API
  - "fallback" -> Returns mock responses (no LLM needed)

10. RUNNING THE PROJECT
================================================================================

Prerequisites:
  - Python 3.14+
  - Node.js 22+
  - Ollama (with model pulled: gemma3:1b)

10.1 Backend
  cd CODEBUDDY/backend
  python -m venv venv
  .\venv\Scripts\activate      (Windows)
  pip install -r requirements.txt
  uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload

  Backend runs at: http://localhost:8000
  API Docs at:     http://localhost:8000/docs

10.2 Frontend
  cd CODEBUDDY/frontend
  npm install
  npx vite --port 5173

  Frontend runs at: http://localhost:5173

10.3 Ollama (required for AI responses)
  ollama serve
  ollama pull gemma3:1b

================================================================================

  cd CODEBUDDY/frontend
  npx vite build              # Output in frontend/dist/

  The build produces ~1MB JS (119 modules) with syntax highlighting.

Deployment Notes:
  - Backend requires Python 3.14+, Ollama running locally
  - CORS_ORIGINS in .env must match frontend URL
  - For production, set a strong SECRET_KEY
